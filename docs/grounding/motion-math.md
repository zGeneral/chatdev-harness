# Motion math — for code-driven movement (the `programmer` role)

Drop-in motion guidance for any animated code the programmer writes — `pygame-ce` `Sprite.update()`
(frame-based) and web CSS/JS alike. Autogenerated movement defaults to stiff `+5 to x each frame`;
this turns that into motion that reads as *alive*.

> Extracted (distilled, not installed) from `dylantarre/animation-principles` (MIT) — Disney's 12
> principles. The numeric heuristics (deform %, frame/ms tables, 1:2 anticipation, 2–4f stagger,
> 10–50% arc height, the volume-preserving squash keyframe `scaleX(1.2)/scaleY(0.8)≈1`) are the repo's;
> the easing-function bodies are the standard implementations of the curves it names.

## RULE 0 — never linear translation
Linear motion "feels robotic and sterile." Every move/scale/fade goes through an easing curve, and
position follows an **arc**, not a straight constant-velocity line. *Only* exceptions: progress bars,
continuous spinners/loops.

## Easing (`t` = progress 0..1 → eased 0..1; pick by intent)
- `ease_out` — **DEFAULT** (≈70–80% of UI; entrances, anything responding to the player):
  `easeOutCubic = 1 - (1-t)**3`
- `ease_in` — exits/departures, impact "punch": `easeInCubic = t**3`
- `ease_in_out` — autonomous/cinematic/elegant: `easeInOutSine = -(cos(pi*t) - 1)/2`  (CSS `cubic-bezier(0.42,0,0.58,1)`)
- `overshoot` (snappy pop / follow-through): `easeOutBack = 1 + 2.70158*(t-1)**3 + 1.70158*(t-1)**2`
- `spring`/damped (settle with micro-bounce): `pos = target + (start-target)*exp(-k*t)*cos(w*t)`, e.g. k≈8, w≈12

Frame usage: keep `self.t` (0..1), `self.t += 1/(fps*duration_s)` each frame, clamp to 1, then
`val = start + (end-start)*ease(self.t)`.

## Durations (pick, don't guess)
micro/feedback 100–200ms · buttons 150–300ms · **standard transition 200–300ms (default)** · complex/page
300–500ms · dramatic 500–1000ms. At 24fps: snappy 4–6f, natural 8–12f, weighty 16–24f (×2.5 at 60fps).
Heavy objects start *and* stop slow. Contrast matters: a fast beat reads fast only after a slow one.

## Squash & stretch (volume-preserving — the #1 polish)
On deform, scale one axis up and the other down so `scaleX*scaleY ≈ 1.0` (the object never appears to
grow/shrink). Impact squash: `sy = 1 - a; sx = 1/sy` (a≈0.10 energetic, 0.02–0.03 subtle, 0.01–0.02 for
"rigid"). Button tap: `sy = 0.95–0.98`, `sx = 1/sy`. Stretch along velocity: `sx = 1+b; sy = 1/(1+b)`.
In pygame apply via `pygame.transform.scale` from these factors and **pin the contact edge** (keep the
bottom from floating).

## Anticipation, arcs, follow-through, exaggeration
- **Anticipation:** any action >200ms gets a wind-up = 30–50% of the main action (classic 1:2 ratio):
  move slightly *opposite* first (+ a squash), then launch.
- **Arcs:** position follows a curve. Cheap arc A→B: `pos = lerp(A,B,ease(t)); pos.y -= arc_h*4*t*(1-t)`
  (parabola, peak at t=0.5). Arc height 10–20% of travel (subtle) up to 30–50% (dramatic).
- **Follow-through / overlap:** trailing parts lag the root 2–4 frames @24fps per hierarchy level (≈0.05s,
  0.10s); end with ease-out + slight overshoot, then a smaller counter-overshoot, then rest.
- **Exaggeration:** a touch of overshoot on the key pose (e.g. peak scale 0.8/1.3, still ~volume-preserving)
  reads as alive. Don't over-ease — too-soft motion feels sluggish/unresponsive.
